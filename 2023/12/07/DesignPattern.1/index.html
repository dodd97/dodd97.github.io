设计模式学习笔记(一)

1 设计模式介绍

  • SimUDuck 父类Duck和子类
  classDiagram 
	class Duck {
		makeSound()*
		swim()*
		display()*
		fly()*
		//...()
	}
	class MallardDuck {
		display()		// 是绿头
	}
	class RubberDuck {
		display()		// 是橡皮
		new fly()		// 橡皮鸭不会飞,需要屏蔽掉父类的抽象方法
	}
	class WoodenDuck {
		display()		// 是木头
		new makeSound()	// new关键字是C#屏蔽父类方法的语法
		new fly()		
	}
	Duck <|-- MallardDuck
	Duck <|-- RubberDuck
	Duck <|-- WoodenDuck

  • 父类的变更会在不经意间影响子类

  • 设计原则: 识别应用中变化的方面,把它们和不变的方面分开

  • 设计原则: 针对接口编程,而不是针对实现编程

  classDiagram 
namespace performFly{
	class IFlyBehavior{
        fly()*
    }
    class FlyWithWings{
        fly()
    }
    class FlyNoWay{
        fly()
    } 
}
<<Interface>> IFlyBehavior
IFlyBehavior <|.. FlyWithWings
IFlyBehavior <|.. FlyNoWay
namespace performMakeSound{
	class IMakeSoundBehavior{
        makeSound()*
    }
    class Quack{
        makeSound()
    }
    class Squeak{
        makeSound()
    } 
    class Mute{
        makeSound()
    }
}
<<Interface>> IMakeSoundBehavior

IMakeSoundBehavior <|.. Squeak
IMakeSoundBehavior <|.. Mute

  • 这种实现方法会让其他类型的对象可以复用fly和makeSound方法,因为行为不再被限制在类里面.也可以直接新增行为,而不修改任何已有的行为类

  • 如果给一只木头鸭增加了一个推进器,让它拥有了飞行的能力.

    • 给Duck新增换flyBehavior的方法
    • 新增行为类

  • 设计原则: 优先使用组合而不是继承

    • HAS-A 比 IS-A 好

  • 以上,是 策略模式

    • 策略模式定义了一个算法族,分别封装起来,使得他们之间可以相互变换,策略让算法的变化独立于使用它的客户
